# FolioFrame Demo Walkthrough Script

## Purpose

Use this script to walk a buyer through FolioFrame as a static service website and static portal/demo surface.

Do not present the demo as live software with active billing, production auth, customer records, monitoring or integrations.

## Suggested Setup

Open the following routes:

- `/`
- `/signature-launch`
- `/pricing`
- `/app`
- `/app/implementation-docs`

## Opening Talk Track

FolioFrame is a buyer-access reliability system for digital product sellers. The core problem is that checkout, access, delivery, onboarding and support often live across disconnected tools. FolioFrame packages the work of mapping, testing and documenting that checkout-to-customer route.

This build is buyer-ready as a static service website and static portal demo. It is not being represented as a live SaaS yet.

## Route 1: Homepage

Point out:

- The positioning: checkout-to-customer systems that are mapped, tested and documented.
- The outcome strip from payment to reporting.
- The premium brand and approved visual assets.
- The safe scope boundary that avoids impossible promises.

Suggested line:

FolioFrame is not promising platform outcomes. It is selling clarity, operational review, delivery safeguards and owner-approved documentation.

## Route 2: Signature Launch

Point out:

- Signature Launch Premium as the high-ticket implementation offer.
- Included modules: FolioFix, DeliveryProof, AccessGuard, AccessDesk, RevenueFrame, AccessSafe, EntitlementFrame, InboxGuard, AccessHub, ProofMetrics and the FolioFrame Verified Delivery Standard.
- The distinction between launch readiness and live monitoring.

Suggested line:

This is the offer page a buyer could deploy first if they want to sell FolioFrame as a premium service before building full SaaS operations.

## Route 3: Pricing

Point out:

- Premium offer hierarchy.
- Lower-ticket audit entry point.
- Retainer path through AccessGuard + ProofMetrics.
- Static pricing boundary: no payment processing is active.

Suggested line:

The pricing architecture supports both implementation revenue and ongoing review work, but checkout collection is intentionally not wired in this static demo.

## Route 4: `/app` Portal Demo

Point out:

- Static/demo notices.
- Command-centre overview.
- Workspace review materials.
- Module status and owner action queue.
- Approved asset usage.

Suggested line:

This is a portal concept and sales demo. It shows how a buyer might review module status, owner actions and handover materials, but it does not display real customer data.

## Route 5: Implementation Docs Hub

Point out:

- Production integration blueprint.
- Webhook event map.
- Client onboarding SOP.
- DeliveryProof test plan.
- LaunchRoom and MigrationFrame runbooks.
- AgencyFrame playbook.

Suggested line:

The implementation docs turn the brand from a website into a service delivery method.

## Disclosure Moment

State clearly:

FolioFrame is ready as a static service website and static demo. It is not ready as a live SaaS until production auth, secure credentials, database configuration, integrations, backend workflows and legal/privacy review are completed.

## Closing Buyer Prompt

The buyer decision is whether to:

- Deploy as a static service brand.
- Start selling the premium implementation manually.
- Build the live SaaS layer.
- Combine service revenue with progressive productization.
