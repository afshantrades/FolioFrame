# FolioFrame Asset Transfer Schedule

## Purpose

This schedule lists the asset classes expected in a FolioFrame closing handover.

It is not a legal assignment instrument. Buyer and seller should use it as a checklist for transfer confirmation.

## Repository Assets

Transfer package should include:

- Application source code.
- `app/` routes and layouts.
- `components/` shared interface components.
- `content/` module, offer, demo and implementation data.
- `docs/` documentation and QA evidence.
- `public/assets/` approved brand assets.
- Prisma schema and live-product foundation files.
- Package manifests and configuration files.

## Curated Motifs

Curated motifs are stored under:

- `public/assets/curated_motifs/`

Use them only as subtle supporting decoration. Do not treat motif files as logo replacements or final brand marks.

## Documentation Assets

Documentation to transfer includes:

- Implementation blueprints.
- SOPs and runbooks.
- QA reports.
- Deployment notes.
- Sale-room package.
- Closing package.
- Static/demo boundary disclosures.

## Excluded Or Restricted Items

Do not transfer or represent as included:

- `.env`
- `.env.local`
- Production credentials
- Local database volumes
- Real customer data
- Rejected generated assets
- Image-generated closing-package page images as final sale materials

## Transfer Confirmation

Buyer should confirm:

- All approved asset files exist.
- No rejected generated asset filenames are referenced.
- No live secrets are included.
- Repository history and branch state are understood.
- Any required domain, trademark or external account transfer is documented separately.
