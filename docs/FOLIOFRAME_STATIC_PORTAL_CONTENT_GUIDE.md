# FolioFrame Static Portal Content Guide

This guide explains how static internal portal content should be written and maintained.

## Portal Rule

Every `/app` route must remain clearly marked as static/demo only until real authentication, data handling, and production integrations are approved.

The current portal must not imply:

- real accounts
- real billing
- real customer records
- real email provider records
- real automation platform records
- live monitoring
- production credentials

## Approved Portal Language

Use operational wording:

- mapped
- tested
- documented
- owner-approved
- monitoring recommended
- delivery safeguards
- practical recovery route
- access lifecycle rules
- operational reporting

## Demo Data Rules

- Use fictional labels such as `Demo client A` or `ENT-DEMO-001`.
- Do not use real names.
- Do not use real email addresses.
- Do not use real transaction records.
- Do not use real platform IDs.
- Do not paste customer support conversations.
- Do not present sample metrics as production performance.

## Module Workspaces

Each module workspace should show:

- portal headline
- portal body
- problem solved
- widgets
- QA checks
- deliverables
- owner actions
- boundaries
- safe claims
- static data preview count

## Owner Review Content

Owner actions should describe practical review steps:

- approve access rules
- review message copy
- confirm product inventory
- assign support or operations owners
- approve handover packets
- confirm production implementation decisions

## Logo Boundary

The portal must continue using the temporary text-only `BrandLockup`. Final logo work remains blocked until an approved production master is available.
